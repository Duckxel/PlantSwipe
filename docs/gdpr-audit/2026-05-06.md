## New Data Surfaces

### 1. `aphydle_buffer_schedule` table (PR #1506)

- **What data is stored:** Singleton configuration row — `id` (text, always `'default'`), `enabled` (boolean), `days_of_week` (text[]), `publish_time_local` (text), `run_time_local` (text), `timezone` (text), `organization_id` (text), `channel_ids` (text[]), `last_run_at` (timestamptz), `last_run_for_date` (date), `last_run_status` (text), `last_run_results` (jsonb), `updated_at` (timestamptz)
- **Where it is stored:** Supabase PostgreSQL — `public.aphydle_buffer_schedule`
- **User linkage:** **None.** No `user_id` or `auth.users` reference. Admin-only via RLS policy `aphydle_buffer_schedule_admin_all` gated by `is_admin_user()`. Anon access revoked.
- **Nature:** Operational config for the automated Buffer social media scheduler. Contains only platform settings (schedule times, Buffer org/channel IDs, execution metadata). No end-user PII.

### 2. Buffer API — new third-party service (PR #1506)

- **Service:** Buffer (api.buffer.com/graphql)
- **Endpoints used:** `GET /api/admin/buffer/organizations`, `GET /api/admin/buffer/channels`, `POST /api/admin/buffer/post`
- **All endpoints admin-only** (protected by `ensureAdmin` middleware)
- **Data sent to Buffer:** Social card images (PNG/JPG of plant content cards), post text (plant facts, captions). No end-user PII is sent.
- **Data received from Buffer:** Organization metadata, channel lists. Stored only in-memory for admin UI rendering, not persisted.
- **API key storage:** Environment variable `BUFFER` — not stored in database.
- **Transient storage:** Card images uploaded to Supabase storage (`adminUploadBucket/buffer-posts/`) for Buffer ingestion. UUIDs as filenames, no user identification.

### 3. `plant_history.field` value expansion (PR #1505)

- **Not a new data surface.** The existing `plant_history` table (already audited in 2026-04-29 report) now stores additional `field` values like `translation:en`, `translation:fr`, etc. No new columns, no schema changes. The `summary` column gains richer annotations (e.g., "Updated FR translation — name, variety, origin") but remains auto-generated from static field labels with no PII.
- **Existing coverage confirmed:** Export handler (server.js line ~21121, item 29) and deletion handler (server.js step 14o) already cover `plant_history` by `author_id`.
