## New Data Surfaces

### 1. `plant_dump_groups` table (PR #1509)

- **What data is stored:** `id` (uuid), `name` (text), `plant_id` (text FK → plants), `created_by` (uuid FK → auth.users ON DELETE SET NULL), `created_at`, `updated_at`
- **Where it is stored:** Supabase PostgreSQL — `public.plant_dump_groups`
- **User linkage:** `created_by` references `auth.users(id)` with `ON DELETE SET NULL`
- **Access control:** Admin-only — RLS policy `pdg_admin` gated by `is_admin_user()`. Anon access revoked. Write privileges revoked from `authenticated` role (service-role only via server.js).
- **Nature:** Staging grouping for bulk plant image uploads. Groups are created by admin/editor users to organize images before promotion to the plant catalog.

### 2. `plant_dump_images` table (PR #1509)

- **What data is stored:** `id` (uuid), `bucket` (text), `path` (text), `url` (text), `original_name` (text), `size_bytes` (integer), `group_id` (uuid FK → plant_dump_groups), `plant_id` (text FK → plants), `uploaded_by` (uuid FK → auth.users ON DELETE SET NULL), `uploaded_at` (timestamptz), `status` (text: 'pending'|'submitted'|'deleted'), `submitted_at`, `deleted_at`
- **Where it is stored:** Supabase PostgreSQL — `public.plant_dump_images`; file objects in Supabase storage bucket `PLANTS` under `plants/dump/` prefix
- **User linkage:** `uploaded_by` references `auth.users(id)` with `ON DELETE SET NULL`
- **PII considerations:** `original_name` contains the filename from the uploader's device (e.g., `IMG_1234.jpg`, `rose_photo.webp`). While typically non-identifying, device filenames can occasionally contain personal information.
- **Access control:** Admin-only — same RLS + privilege restrictions as `plant_dump_groups`.
- **Nature:** Individual staging images of plants uploaded by admins for bulk processing. Images are plant reference photos, not personal user content.

### Deletion Gaps

#### 1. `plant_dump_images.original_name` — NOT cleared during account deletion

- **Obligation:** Right to Erasure
- **Severity:** LOW — Filenames are typically non-identifying (e.g., `IMG_1234.jpg`), but could occasionally contain personal information
- **Details:** While the FK ON DELETE SET NULL correctly severs the `uploaded_by` link when the auth user is deleted, the `original_name` field (device filename) persists. In edge cases, filenames could contain identifying information (e.g., `johns_garden_rose.jpg`).
- **Suggested fix:** Add a step in the GDPR deletion handler to null out `original_name` for the user's dump images:
  ```sql
  UPDATE public.plant_dump_images SET original_name = NULL WHERE uploaded_by = $userId
  ```
  This must run **before** step 18 (auth user deletion), because after the FK cascade fires, `uploaded_by` will be NULL and the rows become unidentifiable.
- **File:** `plant-swipe/server.js` — add before step 18 in `POST /api/account/delete-gdpr`

#### 2. `plant_dump_images` storage objects — NOT deleted during account deletion

- **Obligation:** Right to Erasure
- **Severity:** LOW — Images are of plants, not personal photos
- **Details:** When a user account is deleted, staging images in `PLANTS/plants/dump/` uploaded by that user remain in object storage. The FK cascade nullifies `uploaded_by`, but the storage files persist.
- **Flagged for human review:** Since dump images are plant reference photos (not personal user content), determine whether storage deletion is required. If the images have been promoted to `plant_images` (status='submitted'), the storage object now serves the shared plant catalog and should not be deleted. Only 'pending' or 'deleted' status images could be candidates for removal.
- **File:** `plant-swipe/server.js` — add before step 18 in `POST /api/account/delete-gdpr`
