## Security / Policy Audit

### ✅ Good
- **`aphydle_buffer_schedule` — well-scoped policy and grants.**
  - RLS enabled (`22_aphydle_buffer_schedule.sql:35`).
  - One policy `aphydle_buffer_schedule_admin_all`, scoped `FOR ALL TO authenticated USING (public.is_admin_user((select auth.uid()))) WITH CHECK (public.is_admin_user((select auth.uid())))` — read and write equally gated, ownership keyed on admin status.
  - Explicit `REVOKE ALL FROM anon`, `REVOKE INSERT, UPDATE, DELETE FROM authenticated`, `GRANT SELECT TO authenticated` — defence-in-depth even if a later schema-level grant fires.
  - Service-role traffic from `server.js` bypasses RLS as expected.
  - Singleton-row design (`id text primary key`, pinned to `'default'`) means there is no per-user data and no GDPR purge surface.
- **All three new tables have RLS enabled** — no surface was left at the default `disabled` state.
- **Endpoints are gated server-side.**
  - All `/api/admin/plant-dump/*` routes call `ensureEditor(req, res)` (`server.js:6911, 7000, 7056, 7095, 7132, 7180, 7344, …`) before touching the dump tables.
  - All `/api/admin/aphydle-schedule*` routes call `ensureAdmin(req, res)` (`server.js:5621, 5651, 5690`).
  - The runner gate (`isAphydleAutomationRunner()` + `VITE_SERVER_NAME=MAIN`) prevents the every-minute cron from double-posting from dev mirrors.
- **`plant_images.source` is constrained** to `('uploaded', 'dump', 'web')` via `plant_images_source_check`. No free-text origin; no risk of poisoned `source` values.
- **No new `SECURITY DEFINER` functions** were introduced this week.
- **No new storage buckets** were created. The Plant Image Dump reuses the existing `PLANTS` bucket under the prefix `plants/dump/`, which inherits the bucket's existing storage policies. No widening of `storage.objects` access.

### ⚠ Review
- **`plant_dump_images` / `plant_dump_groups` storage object lifecycle.** On submit, server.js moves `PLANTS/plants/dump/{uuid}.webp` to `PLANTS/plants/images/{plantName}/...` and inserts into `plant_images`. On group delete (`server.js:7423`), the row is removed but the **storage object cleanup happens elsewhere and was not exhaustively walked in this audit**. Confirm with a manual run that abandoned dump objects (uploaded but never submitted) are cleaned up; if not, add a daily cron to delete `status='deleted'` and stale `status='pending'` rows older than 30 days plus their corresponding storage paths.
- **`aphydle_buffer_schedule.last_run_results` JSONB content not constrained.** This column receives per-run output that includes Buffer channel IDs and may include error messages with arbitrary strings from the upstream API. None of those are user PII, but they are operator-attributable; future GDPR audits should account for them when the audit scope widens to "admin activity logging".
- **Migration-timestamp collision** (see Supabase Health) — still flagged because `20260422000000_prevent_self_admin_escalation.sql` is the security-critical half of the collision pair.

### ❌ Problems
- **`POLICY_TOO_BROAD` — `plant_dump_images` (CRITICAL).** Policy `pdi_admin` at `sync_parts/23_plant_dump.sql:43-44` is `FOR ALL TO authenticated USING (true) WITH CHECK (true)`. Any authenticated user — not just admins/editors — can SELECT / INSERT / UPDATE / DELETE rows in `plant_dump_images` directly via PostgREST. The HTTP layer gates the admin UI with `ensureEditor`, but the table policy doesn't enforce that — meaning a regular signed-in user can read every dump image's path / `uploaded_by` / `original_name` and exfiltrate or delete staging rows. Additionally there is **no `REVOKE`** to keep schema-level grants from making this worse.
- **`POLICY_TOO_BROAD` — `plant_dump_groups` (CRITICAL).** Same shape: policy `pdg_admin` at `sync_parts/23_plant_dump.sql:47-48` is `FOR ALL TO authenticated USING (true) WITH CHECK (true)`. Any authenticated user can rename groups, reassign their `plant_id`, or delete them — which cascades into `plant_dump_images.group_id = NULL` and can scramble in-flight admin work.
  - **Recommended fix (both tables):** replace `using (true) with check (true)` with `using (public.is_admin_user((select auth.uid()))) with check (public.is_admin_user((select auth.uid())))`, matching the buffer-schedule pattern. Optionally add `REVOKE ALL ON public.plant_dump_images FROM anon` + `REVOKE INSERT, UPDATE, DELETE ON public.plant_dump_images FROM authenticated` (and likewise for `plant_dump_groups`) so a later schema-level `GRANT` cannot silently widen access. Then add the same to the corresponding migration; no migration was filed for `23_plant_dump.sql` at all yet (only the sync_parts file landed), so this is the right moment to add one.
- **`SYNC_MISSING` — `aphydle_buffer_schedule` not in `allowed_tables`** (also called out under Repo Sync Health; restating here because the consequence is a security regression: a sync run loses the singleton config and the next time the runner ticks it will silently re-`create table if not exists`, ending up with the **default `enabled=false` row** so all scheduled posts stop firing without any operator notification).
- **`MIGRATION_MISSING` — `23_plant_dump.sql` has no companion in `plant-swipe/supabase/migrations/`.** `22_aphydle_buffer_schedule.sql` correctly ships a paired `20260505000000_add_aphydle_buffer_schedule.sql` migration; `23_plant_dump.sql` does not. The commit message for `b6fc407` says "migration applied" but the file is not in-tree, so a fresh `supabase db push` will not create the table on a clean environment. The runtime server.js code does **not** lazily create these tables (unlike `aphydle_buffer_schedule`), so the schema would be missing entirely on a new environment until an operator runs the sync.
