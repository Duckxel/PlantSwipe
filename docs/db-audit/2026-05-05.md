### ⚠ MIGRATION_HISTORY_MISMATCH (carried over, **not** introduced this week)
The previous audit (`docs/db-audit/2026-04-28.md`) flagged a timestamp collision between two migrations that **share the prefix `20260422000000`**:

- `plant-swipe/supabase/migrations/20260422000000_migrate_admin_commentary_to_notes.sql` — landed on `main` 2026-04-23 via PR #1448.
- `plant-swipe/supabase/migrations/20260422000000_prevent_self_admin_escalation.sql` — landed on `main` 2026-04-26 via PR #1459 (one day **before** last week's audit was written, so the audit was right that the second file was problematic but slightly off about which branch had merged it).

Both files are still on `main` today with the same `20260422000000` prefix. Supabase tracks applied migrations by `version` (the timestamp prefix), so a fresh `supabase db push` against an empty environment will register only one of them and silently skip the other — leaving the production schema correct only because both were already applied via the prior route. Action: rename one file to a distinct timestamp (e.g. `20260422000003_prevent_self_admin_escalation.sql`) and, if Supabase already has the original version recorded, run `supabase migration repair --status applied <new-version>` to keep the history table consistent. Do **not** silently delete the row from `supabase_migrations.schema_migrations`.

The collision did **not** originate this week, so no PR from this audit window is responsible — but it is still on `main` and should be resolved before the next migration is added to avoid hitting `version` conflicts on a clean restore.

## Security / Policy Audit

### ✅ Good
- **No new RLS-bearing surfaces this week.** The two new admin AI endpoints (`/api/admin/ai/plant-historical-fact`, `/api/admin/ai/plant-export-content`) are gated by `ensureEditor(req, res)` on the server (server.js:5221, 5329), which validates the Supabase bearer token and confirms the caller has `is_admin = true` or role `admin`/`editor`. They write only into `ai_usage_events` via the existing service-role `logAiUsage()` helper.
- **`plant_history` / `plant_admin_notes` GDPR anonymisation** uses the service-role Supabase client / `postgres-js` `sql` connection. RLS is bypassed by the service role, so no extra policy work is required. The new SQL on `plant_admin_notes` is a scoped `UPDATE … WHERE author_id = $userId` that cannot leak to other users' rows.
- **`plant_history` retention** — the existing `ON DELETE SET NULL` on `author_id` (sync_parts/03_plants_and_colors.sql:2222) already preserves the audit trail while detaching the deleted user. The new GDPR step 14o just records a count for the audit log; it does not loosen any access path.
- **Sentinel email-campaign-runner fix** replaces `Math.random()` with `crypto.getRandomValues()` for tracking-pixel ID entropy. No grants / RLS / SQL touched. Safe.
- **Aphydle analytics endpoints** are gated server-side by `ensureEditor` (admin/editor only). They run aggregate queries against an isolated `aphydle.*` schema on a separate connection and never expose row-level PII.
- **`gardens.ts` upsert fix** — purely a PostgREST `onConflict` encoding correction, aligning with the existing unique index. No security implication.
